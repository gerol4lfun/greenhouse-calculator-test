### Логика генерации КП

#### Короткое КП (Short CP)
- **Файл:** `js/scripts.js`
- **Функция:** `generateShortOffer(finalTotalPrice1, selectedEntry)` (строки 2147-2272)
- **Где вызывается:** В `performCalculation()` после расчёта (строка 1322)
- **Формат:**
  ```
  {ФОРМА} теплица {ШИРИНА}×{ДЛИНА} [с доставкой до {АДРЕС}] [со сборкой/в комплекте]
  
  1) Стандарт: {ИТОГО_1} ₽
  2) С запасом по нагрузке: {ИТОГО_2} ₽ ({ПРИЧИНА_ЗАПАСА})
  
  Без предоплаты. Гарантия 15 лет. Бесплатная заморозка стоимости.
  [🎁 Вам доступен подарок.] (если >= 35000)
  
  Ближайшая дата доставки — {ДАТА}.
  ```
- **Вывод:** В `textarea#commercial-offer-short`

#### Длинное КП (Long CP)
- **Файл:** `js/scripts.js`
- **Функция:** `generateCommercialOffer(...)` (строки 1484-1648)
- **Где вызывается:** В `performCalculation()` после расчёта (строка 1319)
- **Формат:** Полное описание теплицы (название, каркас, размеры, характеристики, цены, второй вариант если есть)
- **Вывод:** В `textarea#commercial-offer` (вкладка "Длинное КП")
- **Инвариант (платные допы и подарки):** платный доп обязан входить в цену, отображаться в составе/длинном КП и не удаляться логикой подарков. Подарки добавляются только в блок 🎁; `rebuildLongOfferWithGifts` чистит только блоки с 🎁, не трогая платный состав.

#### Текст заказа (полный шаблон для клиента)
- **Файл:** `js/scripts.js`
- **Функция:** `generateFullOrderTemplate(calc, client, orderCart)` — приветствие (с `displayName` или «Клиент»), ВАШ ЗАКАЗ: при 2+ одинаковых теплицах — одна позиция с множителем (х2/х3) и одна «Итоговая стоимость» (`buildGreenhouseBlockIdentical`); при 2+ разных — блоки «1 теплица», «2 теплица», … + стоимость доставки + «Общий итог заказа». Подвал: оплата, контакты, «О доставке»; блок «О сборке» только при выбранной сборке (`ORDER_FOOTER_BASE` + при `calc.assemblyText` — `ORDER_FOOTER_ASSEMBLY`).
- **Превью до отправки:** при наличии корзины `updateClientOfferFromShort()` вызывает `generateFullOrderTemplate(..., orderCart)`; `updateOrderCartUI()` в конце вызывает `updateClientOfferFromShort()` — текст во вкладке «Текст заказа» до «Оформить заказ» совпадает с итоговым и обновляется при изменении корзины.
- **Где используется:** вкладка «Текст заказа» (превью до и полный шаблон после «Оформить заказ»).

#### Выбор альтернативного варианта (upsell)
- **Файл:** `js/scripts.js`
- **Функция:** `pickReserveVariant()` (строки 1900-1999)
- **Логика:**
  1. Если шаг дуг != 0.65 → предлагает шаг 0.65
  2. Если шаг 0.65 и каркас 20×20 → предлагает 40×20
  3. Если каркас 20×20+20×20 → предлагает 40×20+20×20
  4. Если каркас максимальный → улучшает поликарбонат (Стандарт → Люкс → Премиум)
- **Функция расчёта альтернативы:** `computeFinalTotalPriceForVariant(overrideParams)` (строки 2001-2145)
- **Используется в:** `generateShortOffer()` (строка 2212) и `generateCommercialOffer()` (строка 1621)

### Логика дат доставки (delivery_calendar)

**Source of truth по логике дат и UI календарей — этот раздел.**

**Правила доступности дат:**
- **delivery_calendar** — слой ограничений (запрещённые даты), не полный список разрешённых.
- **Прошлые даты и сегодня** — считаются по Москве (UTC+3), недоступны.
- **Будущая дата + явный запрет** — недоступна.
- **Будущая дата + нет запрета** — доступна.
- Логика одинакова для всех направлений.
- Одинаковая логика используется в основной форме и в edit modal.
- **nearest/default date** — первая будущая незапрещённая дата.
- **Edit modal:** должен использовать канонический city key (из адреса), а не региональную строку.
- **Fallback:** при отсутствии данных в delivery_calendar используется delivery_dates (пока остаётся).
- **Месяцы без загруженных ограничений** — по бизнес-логике считаются доступными.

**Статус загрузки ограничений (14.03.2026):**
- Март загружен.
- Апрель и май планируются.
- Будущие месяцы без ограничений считаются доступными.

**UI (после этапа delivery_calendar, 14.03.2026):**
- Календарь основной формы работает по новой логике.
- Edit modal выровнен по той же логике.
- Модалка «Актуальные даты доставки» — split-view: слева список городов, справа календарь выбранного города. Цветовые состояния: серый (прошлое), зелёный (доступно), жёлтый (ограничено), красный (запрещено). Использует новую архитектуру дат, а не старую модель «одна дата на город».
- Блок доставки после расчёта адреса: стоимость + заголовок «Даты доставки» + плашки «Без сборки — 17 марта» / «Со сборкой — 20 марта» (формат без года). Активная строка подсвечивается по чекбоксу сборки.
- Короткое и длинное КП используют дату по выбранному режиму сборки (чекбокс «Сборка»).
- Fallback «17 февраля» удалён; при отсутствии даты строка не добавляется.

## EDIT FLOW ЗАКАЗОВ — ИНВАРИАНТЫ

- **Commercial offer при edit:** для пересчитанных позиций источник данных — результат расчёта и item в editOrderComposition. Старый commercial_offer — только fallback для нетронутых позиций.
- **Gifts = slots:** подарки трактуются как слоты; slot count зависит только от total; выбор подарка внутри слота меняет только содержимое слота, не tier.
- **Gift slots в edit/add panel:** slot count меняется только от total; при tier-change — заметный toast.
- **Cancelled заказ:** нельзя редактировать. Защита на UI (нет кнопки) и перед save.
- **Платные допы:** не должны пропадать из long КП из-за логики подарков. Edit платных допов: edit → save → Supabase → карточка → commercial_offer.
- **После изменений edit flow** проверять: create, edit даты, edit состава, edit платных допов, cancel, попытка редактирования cancelled.

   - Кнопка "Добавить грядки" в интерфейсе
   - Модальное окно с двухэтапным выбором:
     - Шаг 1: Выбор типа (Низкие 19см / Высокие 38см) с изображениями
     - Шаг 2: Выбор характеристик (ширина, длина) с учетом размеров теплицы
   - Фильтрация по длине теплицы (показываются только грядки не длиннее теплицы)

2. **Рекомендуемый вариант грядок**
   - **Для теплиц 4м:** 2 боковые грядки по 1м шириной (без центральной)
   - **Для теплиц 6м+:** 
     - Центральная грядка: 4м длиной (ширина зависит от ширины теплицы)
     - Боковые грядки: длина = длина теплицы (до 12м)
     - Ширина зависит от ширины теплицы:
       - Если ширина >= 3.5м: центральная 0.8м, боковые 1м
       - Иначе: центральная 0.5м, боковые 0.8м
   - Два варианта добавления: "Без сборки" и "Со сборкой"
   - Автоматическое закрытие модального окна после выбора

3. **Цены на грядки**
   - Данные в массиве `BEDS_DATA` (40 вариантов)
   - Цены одинаковые для всех городов (захардкожены)
   - Ширины: 0.5м, 0.65м, 0.8м, 1м
   - Длины: 4м, 6м, 8м, 10м, 12м
   - Высоты: 19см (низкие), 38см (высокие)

4. **Сборка грядок**
   - Опциональная услуга с ценами по длине теплицы (4м: 990, 6м: 1490, 8м: 1990, 10м: 2490, 12м: 2990, 14м: 3490, 16м: 3990)

5. **Интеграция в КП**
   - Грядки добавляются в длинное и короткое КП
   - Учитываются в общей стоимости

6. **Система уведомлений (Toast)**

## БЛОК 5 — СИСТЕМА ПОДАРКОВ (v170-v179)

**Gift logic = slot model.** Источник истины: `docs/GIFT_TRUTH.md`. Любые изменения подарков проверять против него. Не вводить логику fixed bundles без новой бизнес-причины.

**Инварианты подарков:**
- Подарки = слоты (не fixed bundles).
- Логика подарков не должна возвращаться к fixed bundles.
- Каждый свободный слот должен давать базовые варианты: window, drip-mech, stakes-4.
- Автомат для форточки — отдельное зависимое правило (от 55k, при наличии форточки).

### ✅ Реализовано:

1. **Динамическая система подарков**
   - Подарки появляются автоматически при достижении определенной суммы заказа:
     - От 35,000₽: 1 подарок
     - От 55,000₽: 2 подарка
     - От 75,000₽: 3 подарка
   - Блок подарков обновляется автоматически при изменении параметров калькулятора
   - Выпадающие списки для выбора подарков (до 3 слотов)

2. **Доступные подарки**
   - Дополнительная форточка (`window`)
   - Капельный полив механический (`drip-mech`)
   - Автомат для форточки (`window-automation`) - требует наличия дополнительной форточки
   - Автоматическая форточка (форточка + автомат) (`window-auto`) - занимает 2 слота
   - 4 грунтозацепа (`stakes-4`)
   - Опция "Без подарка" для каждого слота

3. **Логика зависимостей**
   - "Автомат для форточки" доступен только если есть дополнительная форточка (купленная или подаренная)
   - "Автоматическая форточка" (комбинированный вариант) занимает 2 слота подарков
   - При выборе "Автоматическая форточка" в первом слоте, второй слот автоматически скрывается

4. **Интеграция в КП**
   - Подарки автоматически добавляются в короткое и длинное КП
   - Динамическое обновление КП при выборе/изменении подарков (без пересчета всей стоимости)
   - Правильное склонение: "Подарок" (1), "Подарка" (2-4), "Подарки" (5+)
   - Подсчет количества одинаковых подарков: если выбран 2 одинаковых подарка, показывается "Дополнительная форточка (2 шт)"

5. **UI элементы**
   - Кнопка "Как работают подарки?" с модальным окном объяснения логики
   - Кнопки "Добавить подарок" и "Удалить подарок" для управления
   - Модальное окно с подробным объяснением системы подарков (загружается из FAQ)

6. **Технические улучшения**
   - Функция `updateGiftsBlock()` - динамическое обновление блока подарков
   - Функция `updateCommercialOffersWithGifts()` - обновление только секции подарков в КП без полного пересчета
   - Функция `getGiftsText()` - формирование текста подарков из DOM элементов

   - Агрессивное удаление всех старых блоков подарков в КП через цикл `while` для предотвращения дублирования

### 🔧 Ключевые функции:
